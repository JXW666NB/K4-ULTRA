## 1. 快速开始

### 启动

直接双击 `源码编辑器4.0.exe` 即可运行。

### 修改代码

1. 用文本编辑器打开 `resources\app\build\kitten.822d814413fb10654fde.js`（核心代码，约10MB）
2. 搜索你要修改的关键词
3. 修改后保存，重新打开exe即可生效
4. **不需要编译、不需要打包、不需要npm**

### 修改前备份

```powershell
Copy-Item "resources\app\build\kitten.822d814413fb10654fde.js" "resources\app\build\kitten.822d814413fb10654fde.js.bak"
```

## 3. 核心架构

### 技术栈

- **Electron** - 桌面应用框架
- **React** - UI框架
- **Redux** - 状态管理
- **Blockly** - 积木编程引擎（Google开源）
- **Webpack** - 模块打包
- **Blink框架** - codemao自研的积木执行引擎（基于Blockly扩展）

### 启动流程

```
双击 源码编辑器4.0.exe
  → Electron加载 main/index.js（主进程）
    → 创建窗口，加载 build/index.html
      → 按顺序加载: runtime.js → vendors.js → kitten.js → extension-loader.js
        → kitten.js 初始化 Blockly + React + Redux + 整个编辑器UI
          → 编辑器显示
```

### 代码结构（kitten.js 内部）

```
kitten.js 包含 341 个模块，每个模块格式：
"模块ID": function(e, t, n) {
    // e = 模块导出
    // t = 模块对象
    // n = require函数（n("模块ID") 加载其他模块）
    "use strict";
    ...
}
```

### 数据流

```
用户操作 → Action → Reducer → Store → React组件 → UI更新
         ↕                              ↕
      Blockly引擎 ←→ 工作区SVG ←→ 积木渲染
```

## 4. 如何修改工具栏分类

### 概念

左侧工具栏的每个分类（事件、控制、运动等）由以下部分组成：

1. **分类定义** - 名称、颜色、图标、包含的积木列表
2. **分类ID** - 唯一标识符（如 `"events"`, `"control"`）
3. **分类排序** - 决定显示顺序的数组
4. **显示状态** - 控制分类是否可见的开关

### 修改已有分类

搜索分类名称（如 `category_name: Blockly.Msg.events`），修改其属性：

```javascript
events: {
    category_name: Blockly.Msg.events,    // 分类显示名
    color: Blockly.Msg.EVENTS_HUE,        // 分类颜色
    icon: a.a.event,                       // 分类图标
    id: s.c.events.id,                     // 分类ID
    blocks: [                              // 包含的积木类型列表
        "start_on_click",
        "sprite_on_tap",
        "on_swipe",
        // ... 添加更多积木
    ]
}
```

### 添加新分类（完整步骤）

## 5. 如何添加自定义积木

### 方法一：注册Blockly积木类型

在 `kitten.js` 中搜索 `Blockly.Blocks` 或 `register_block`，找到积木注册位置。

```javascript
// 注册一个新积木
Blockly.Blocks['my_custom_block'] = {
    init: function() {
        this.appendDummyInput()
            .appendField("我的自定义积木");
        this.setOutput(true, "String");
        this.setColour("#FF6B35");
        this.setTooltip("这是一个自定义积木");
    }
};
```

### 方法二：在分类的blocks数组中引用

注册积木后，把积木类型名加到分类的 `blocks` 数组中：

```javascript
events: {
    // ...
    blocks: ["start_on_click", "my_custom_block", ...]
}
```

### 积木类型定义格式

```javascript
{
    type: "block_type_name",           // 积木类型名（唯一）
    message0: "移动 %1 步",            // 积木显示文本
    args0: [                           // 参数定义
        { type: "input_value", name: "STEPS", check: "Number" }
    ],
    previousStatement: null,           // 上方连接（Statement）
    nextStatement: null,               // 下方连接（Statement）
    colour: "#4C97FF",                 // 颜色
    tooltip: "让角色移动指定步数",     // 提示
    helpUrl: ""                        // 帮助链接
}
```

```javascript
// extension-loader.js 示例
(function() {

        if (typeof Blockly === 'undefined' || !Blockly.getMainWorkspace()) return;

        // 注册积木
        Blockly.Blocks['ext_hello'] = {
            init: function() {
                this.appendDummyInput().appendField("Hello 扩展!");
                this.setOutput(true, "String");

        };

        // 刷新工具栏
        var workspace = Blockly.getMainWorkspace();
        var toolbox = workspace.options.languageTree;
        // ... 注入新分类到toolbox
        workspace.updateToolbox(toolbox);
    }, 1000);
})();
```

## 7. 扩展积木完整开发指南 ★重要★

> 本章是经过实战验证的完整教程。我们成功开发了包含13个系统API积木的扩展插件，
> 踩遍了所有坑，现在把完整流程和经验记录下来。
>
> **适用场景**：你想添加一组有实际功能的扩展积木（不只是显示好看，还要能真正执行功能）。

### 7.1 总体架构理解（必读！）

K4编辑器的积木执行系统是一个**双层架构**，理解这个架构是开发能工作的扩展积木的前提：

```
┌─────────────────────────────────────────────────────────────┐
│                    K4 积木执行双层架构                        │
├─────────────────────────────────────────────────────────────┤
│                                                             │
│  ┌─────────────┐     ┌──────────────┐     ┌──────────────┐ │
│  │ 积木定义     │ ──→ │ JS Generator  │ ──→ │ 代码预览      │ │
│  │ (JSON)      │     │ (代码生成器)   │     │ ("看代码")    │ │
│  └─────────────┘     └──────────────┘     └──────────────┘ │
│         │                                               ▲   │
│         │              ★ 主执行路径 ★                    │   │
│         ↓                                               │   │
│  ┌─────────────┐     ┌──────────────┐     ┌──────────────┐ │
│  │ 积木定义     │ ──→ │ OptiCompiler  │ ──→ │ OptiRunner   │ │
│  │ (JSON)      │     │ (字节码编译器)  │     │ (字节码解释器)│ │
│  └─────────────┘     └──────────────┘     └──────────────┘ │
│                              │                    │        │
│                              ▼                    ▼        │
│                    ┌──────────────────────────────┐        │
│                    │     Registry (单例)           │        │
│                    │  domain_function_index[]      │        │
│                    │  domain_function_list[]       │        │
│                    └──────────────────────────────┘        │
│                                                             │
└─────────────────────────────────────────────────────────────┘
```

**关键概念**：

| 概念 | 说明 |
|------|------|
| **Domain Function** | 积木的运行时实现函数，接收参数并返回结果 |
| **Registry** | 单例注册中心，管理所有domain function的索引映射 |
| **OptiCompiler** | 将积木编译为字节码，通过block.type查找domain function索引 |
| **OptiRunner** | 执行字节码时，用数字索引调用对应的domain function |
| **JS Generator** | 生成可读JS代码字符串，仅用于"看代码"展示，不参与实际执行 |

**重要结论**：要让你的积木真正工作，必须同时完成以下三件事：
1. 定义积木外观（JSON格式）
2. 注册Domain Function（运行时函数）
3. 注册JS Generator（代码生成器，用于"看代码"功能）

### 7.2 开发一个能工作的扩展积木 — 完整步骤

下面以一个实际的例子说明：假设你要添加一个 **"获取当前时间戳"** 的积木。

#### 第一步：添加分类定义（9处修改）

参考第4节的9处修改规范，添加一个新分类。假设分类名为 `my_extension`。

#### 第二步：定义积木外观（JSON Block Definition）

搜索 `define_blocks_with_json_array`，找到积木批量定义的位置（约134139行），添加你的积木：

```javascript
e.define_blocks_with_json_array([{
    // ====== 你的新积木 ======
    type: "my_get_timestamp",               // ★ 积木类型名（全局唯一）
    message0: "获取时间戳",                  // 积木上显示的文字
    args0: [],                               // 无参数
    output: "Number",                        // ★ 输出数值类型
    colour: "%{BKY_SENSING_HUE}",           // 使用侦测类颜色
    inputsInline: !0,                        // ★ 内联输入模式（推荐设为true）
    tooltip: "返回当前Unix时间戳"
}, {
    // 可以继续定义更多积木...
    // ...
}]);
```

**参数类型对照表**：

| 参数用途 | JSON写法 | 说明 |
|---------|----------|------|
| 文本输入 | `{type:"input_value", name:"PATH", check:"String"}` | 连接一个值积木 |
| 下拉选择 | `{type:"field_dropdown", name:"MODE", options:[["打开","open"],["保存","save"]]}` | 固定选项列表 |
| 数字输入 | `{type:"input_value", name:"NUM", check:"Number"}` | 连接一个数值积木 |
| 无参数 | 不写args0或写空数组 | 如上面的时间戳例子 |

**输出/语句类型对照**：

| 用途 | 写法 | 说明 |
|------|------|------|
| 有返回值的积木（值积木） | `output: "String"` / `"Number"` / `"Boolean"` / `"Array"` | 圆形连接口 |
| 无返回值的积木（语句积木） | `previousStatement: null, nextStatement: null` | 方形连接口 |

#### 第三步：注册Domain Function（运行时函数）★★★ 最关键 ★★★

这是让你的积木**真正能执行**的核心步骤！

搜索 `get_domain_functions` 函数（约150140行），在这个函数内部的 `v` 对象中添加你的运行时函数。

**什么是v对象？**

`get_domain_functions` 函数内部有两个大对象：
- **`p` 对象**（约150194行）：包含 year/month/date/hour 等辅助函数，**不是**被注册的domain functions
- **`v` 对象**（约150250行）：包含 when/wait/random/text_join/math_arithmetic 等核心domain functions，**这才是被return并注册的对象**

**你必须把函数注入到 `v` 对象中！注入到p对象不会生效！**

在 `v` 对象末尾（最后一个原生函数之后、`};`之前），添加你的函数：

```javascript
// 在 v 对象中追加你的 domain function
my_get_timestamp: function(e) {             // e = 参数对象
    try {
        var result = Date.now();            // 实际业务逻辑
        console.log('[MyExt] timestamp=' + result);  // 可选调试日志
        return result;                       // ★ 必须return返回值
    } catch(err) {
        console.error('[MyExt] error:', err.message);
        return 0;                            // 出错时返回默认值
    }
},
// 继续添加更多函数...
```

**Domain Function的标准签名**：

```javascript
your_function_name: function(args) {
    // args 是一个对象，key是你在积木定义中设置的参数name
    // 例如 args.PATH, args.CONTENT, args.MODE 等
    try {
        // 你的实际逻辑
        return 结果;    // 值积木必须有return
    } catch(err) {
        return 默认值;  // 不要抛异常！否则会中断程序
    }
}
```

**关于参数访问**：
- 如果积木有一个名为 `PATH` 的输入参数，通过 `args.PATH` 访问
- 如果积木有一个名为 `MODE` 的下拉框，通过 `args.MODE` 访问（得到的是下拉选项的值，如 "open"/"save"）
- 语句积木（无output）不需要return，但建议也return一下

#### 第四步：注册备份Domain Function（Heart IIFE路径）★★ 重要 ★★

搜索 `o.basic_blocks().load_domain_functions()` （约59396行），在它后面的立即执行函数(IIFE)中，添加备份注册。

为什么要两处注册？因为存在两条注册路径，确保至少一条能生效：

```javascript
// 在 load_domain_functions() 后面的 IIFE 中（约59405行起）
var _reg = _bbp.registry;

// ★ 通过require获取原生API（不要用window.FS/window.System！见踩坑记录）
var _my_fs = null;
try { var _nfs = require('fs'); _my_fs = { /* 封装fs方法 */ }; } catch(e){}

var _my_funcs = {
    my_get_timestamp: function(args) {
        try { return Date.now() } catch(e) { return 0 }
    }
    // ... 更多函数
};

for (var _sid in _my_funcs) {
    _reg.register({
        namespace: "",        // ★ 空字符串，表示全局命名空间
        id: _sid,
        domain_function: _my_funcs[_sid]
    });
}
console.log('[MyExt] ' + Object.keys(_my_funcs).length + ' domain functions registered');
```

**注意**：这里的namespace必须是空字符串 `""`。因为编译器查找domain function时用的是积木的type名称（如 `"my_get_timestamp"`），而Registry的namespaced_id规则是：namespace为空时直接返回id。

#### 第五步：注册JS Generator（代码生成器）

搜索 `blink.js_generator.register`（约134343行），为每个积木注册代码生成器。

这个生成器的作用是：当用户点击"看代码"按钮时，生成可读的JavaScript代码字符串。

```javascript
// 为值积木注册generator
gen.register("my_get_timestamp", function(block) {
    // block 是积木实例
    return ['(Date.now())', gen.ORDER_FUNCTION_CALL]
});

// 为语句积木注册generator示例
gen.register("my_do_something", function(block) {
    var path = gen.value_to_code(block, "PATH", gen.ORDER_NONE) || '""';
    return 'console.log(' + path + ');\\n'
});
```

**Generator的返回格式**：

| 积木类型 | 返回格式 | 示例 |
|---------|---------|------|
| 值积木（有output） | `[code_string, order]` | `['(Date.now())', gen.ORDER_FUNCTION_CALL]` |

### 7.4 需要调用Node.js/Electron原生API的情况

如果你的积木需要访问文件系统、剪贴板、系统信息等原生功能：

#### 错误方式（不可行！）

```javascript
// ❌ 这样不行！contextIsolation导致window.FS/window.System对渲染进程不可见
window.FS.exists(path)          // window.FS is undefined
window.System.getOS()           // window.System is undefined
```

#### 正确方式（直接require）

```javascript
// ✅ 正确！绕过contextIsolation限制
var _nodeFs = require('fs');
var _electron = require('electron');

// 文件操作
_nodeFs.existsSync(path)                        // 检查文件是否存在
_nodeFs.readFileSync(path, 'utf8')              // 读取文件
_nodeFs.writeFileSync(path, data, 'utf8')        // 写入文件
_nodeFs.unlinkSync(path)                         // 删除文件
_nodeFs.mkdirSync(path, {recursive:true})        // 创建目录
_nodeFs.readdirSync(path)                       // 列出目录

// Electron API
var _shell = _electron.shell;                   // shell模块
_shell.openExternal(url);                        // 打开外部URL
var _clipboard = _electron.clipboard;            // 剪贴板
_clipboard.readText();                           // 读取剪贴板
_clipboard.writeText(text);                      // 写入剪贴板
var _dialog = _electron.dialog;                  // 对话框
_dialog.showOpenDialogSync({properties:["openFile"]});  // 文件选择对话框
```

**为什么不能用window.FS/System？**

K4的Electron启用了 `contextIsolation: true`，这意味着：
- `preload.js` 中通过 `globalThis.FS = ...` 设置的全局变量
- 对渲染进程（即kitten.js运行的上下文）来说是**不可见的**
- 这是Electron的安全机制，防止渲染进程访问Node.js API

**解决方案**：在kitten.js内部直接使用 `require('fs')` 和 `require('electron')`，因为webpack打包后的代码仍然可以通过CommonJS require访问Node.js内置模块。

## 10. 魔改经验与踩坑记录

### 踩坑1：修改不生效

**现象**：改了代码但重启exe没变化。

**原因**：
- Electron有缓存，需要完全关闭进程（包括后台）
- 修改的文件路径不对（注意是 `resources\app\build\` 不是别的目录）

**解决**：
```powershell
# 强制关闭K4进程
Get-Process -Name "源码编辑器4.0" | Stop-Process -Force
# 然后重新打开
```

### 踩坑2：白屏

**现象**：启动后白屏，控制台有报错。

**原因**：JavaScript语法错误（少逗号、多括号等）。

**解决**：
- 检查修改处的语法
- 恢复备份文件重试

### 踩坑3：分类不显示

**现象**：添加了新分类但左侧看不到。

**原因**：只改了分类定义，没改显示状态。

**解决**：确保修改了全部9处（分类定义x2、枚举x1、ID映射x1、排序数组x1、显示状态x4）。

### 踩坑4：积木拖不出来

**现象**：分类显示了但积木拖不到工作区。

**原因**：积木类型名拼写错误，或积木未注册。

**解决**：确保 `blocks` 数组中的积木类型名与 `Blockly.Blocks` 中注册的完全一致。

### 踩坑5：中文乱码

**现象**：中文显示为乱码。

**原因**：文件编码不是UTF-8。

**解决**：用支持UTF-8的编辑器（VS Code、Notepad++）编辑。

### 踩坑6：extension-loader.js不加载

**现象**：扩展代码不执行。

**原因**：`index.html` 中没有引入该文件。

**解决**：检查 `index.html` 底部是否有 `<script src="extension-loader.js"></script>`。

### 踩坑7：window.FS / window.System 不可用 ★★★ 极其重要 ★★★

**现象**：调试面板显示 `FS=false System=false`，或者积木无输出无报错。

**原因**：K4的Electron启用了 `contextIsolation: true`。preload.js中通过 `globalThis.FS` 和 `globalThis.System` 设置的全局变量，对渲染进程（kitten.js运行环境）**完全不可见**。这是Electron的安全机制。

**错误做法**：
```javascript
// ❌ 这些全是undefined！
window.FS.existsSync(path)
window.System.getOS()
globalThis.FS.readFile(path)
```

**正确做法**：
```javascript
// ✅ 直接require，绕过contextIsolation
var _fs = require('fs');
var _el = require('electron');
_fs.readFileSync(path, 'utf8');           // 文件操作
_el.shell.openExternal(url);              // 打开URL
_el.clipboard.readText();                 // 剪贴板
_el.dialog.showOpenDialogSync({...});     // 对话框
```

**验证方法**：在代码中添加 `typeof require` 检查，如果返回 `"function"` 则说明可以直接使用require。

### 踩坑8：Domain Function注入到错误的对象 ★★★ 极其重要 ★★★

**现象**：积木可见可拖拽，点击运行无报错但也无任何效果。调试面板只有INIT消息，没有REGISTER消息，也没有函数调用日志。

**原因**：`get_domain_functions` 函数内部有两个对象：
- **`p` 对象**：辅助函数（year/month/date等），**不会被注册**
- **`v` 对象**：真正的domain functions（when/wait/random等），**才是被return并注册的对象**

如果你把函数注入到了 `p` 对象中，它永远不会被执行。

**错误做法**：
```javascript
var p = {
    year: function(e) { ... },
    month: function(e) { ... },
    my_new_func: function(e) { ... }  // ❌ 放在了p里面！不会被执行
};
var v = {
    when: function(e) { ... },
    wait: function(e) { ... }
};
return v;  // 只返回v，p被丢弃
```

**正确做法**：
```javascript
var p = { year: ..., month: ... };
var v = {
    when: function(e) { ... },
    wait: function(e) { ... },
    my_new_func: function(e) { ... }  // ✅ 放在v里面！会被注册执行
};
return v;
```

**如何验证**：在 `return function(){...}` 闭包内添加日志，检查你的函数是否出现在 `Object.keys(v)` 的结果中。

### 踩坑9：return v 被误改为 return p ★★★ 致命错误 ★★★

**现象**：修改后所有原生积木报错"调用的函数不存在，联系技术喵或班主任帮你解决"。

**原因**：有人误以为要把 `return v` 改成 `return p` 来让注入的函数生效。这会导致整个运行时崩溃——因为 `p` 对象中没有 when/wait/random 等核心函数，所有原生积木都会找不到对应的domain function。

**解决**：**永远保持 `return v` 不变**。把你的函数注入到 `v` 对象中即可。

### 踩坑10：积木无输出且无报错（静默失败）

**现象**：积木看起来正常运行（不报错），但没有任何效果。值积木返回空/假/零，语句积木什么都不做。

**原因**：通常是因为domain function内部使用了不可用的API（如 `window.FS`），而try/catch吞掉了异常，静默返回了默认值。

**排查步骤**：
1. 确认domain function确实被调用了（添加日志到函数入口）
2. 确认使用的API可用（添加 `typeof` 检查）
3. 检查try/catch是否掩盖了真实错误（临时去掉try/catch测试）
4. 确认Heart IIFE备份注册中也使用了正确的API（不用window.FS）

### 踩坑11：下拉框太窄显示不全

**现象**：积木的下拉选项文字被截断，看不到完整内容。

**原因**：FieldDropdown的最小宽度默认只有20px。

**解决**：搜索 `update_min_width` 方法，将最小宽度从20改为50：
```javascript
t.prototype.update_min_width = function(e) {
    if (!this.text_element) return 50;    // 默认返回50而非20
    if (e >= 50) return e;                // 阈值从20改为50
    // ...
}
```

同时在积木JSON定义中设置 `inputsInline: !0` 使布局更紧凑。

### 踩坑12：缺少inputsInline导致积木排版混乱

**现象**：多参数积木的输入框垂直排列，占用过多空间。

**解决**：在每个积木的JSON定义中添加 `inputsInline: !0`（即 `true`），使参数水平排列。

### 踩坑13：重复注入导致问题

**现象**：多次修改后出现意外行为。

**原因**：多处修改可能重复注册同一个domain function。Registry中首次注册分配索引，后续同名注册会被忽略（因为 `null == this.domain_function_index[c]` 条件）。

**解决**：确保只在一个地方注册，或者利用Registry的去重机制（同key不覆盖）。

### 踩坑14：不知道该修改哪个文件

**现象**：不知道改哪个文件来实现某功能。

**原因**：K4的核心代码都在 `kitten.822d814413fb10654fde.js` 这一个文件里。

**解决**：所有编辑器逻辑（积木定义、执行引擎、UI渲染、代码生成）全在这一个文件中。搜索关键词定位即可。
